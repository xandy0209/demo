# PRD 生成规范 (PRD Generation Guidelines)

当生成产品需求文档 (PRD) 的 Markdown 文件时，必须严格遵循以下格式规范：

## 1. 标题与层级 (Title & Heading Levels)
* **顶部标题**：不要使用一级标题 `#`，必须作为 HTML 注释呈现。
  * 格式：`<!-- [模块名] 需求文档 (PRD) -->`
* **封面内容**：在 HTML 注释之后，正文开始之前，必须包含以下格式的封面内容（根据实际情况替换括号内变量），并在封面内容与正文之间使用 `---` 作为分隔符：
  ```markdown
  内蒙移动政企运维系统
  [需求名称]
  需求文档



  版本：V1.0

  [XXXX]年[xx]月[xx]日

  ---
  ```
* **标题提级**：原有的二级标题提升为一级标题 (`#`)，三级标题提升为二级标题 (`##`)，依此类推。

## 2. 标题编号 (Heading Numbering)
* **去除编号**：生成的 MD 文档中，各级标题必须去掉所有的数字或中文编号前缀。
  * 错误示例：`# 一：需求背景`、`## 1、需求背景`、`### 4.1 数据查询`
  * 正确示例：`# 需求背景`、`## 需求背景`、`### 数据查询`

## 3. 列表层级符号 (List Hierarchy Symbols)
在内容部分，必须按照以下层级顺序使用特定的项目符号或编号，以体现内容的上下层级关系。**注意：在描述界面元素、表格列定义等详细内容时，如果使用 Markdown 原生列表（如 `- ` 或 `* `）或 Markdown 表格能更清晰地表达，则优先使用原生 Markdown 语法，不必强行套用此层级符号。此层级符号主要用于纯文本的层级描述。**
* **第一层级**：使用阿拉伯数字加点，如 `1. `、`2. `、`3. `
* **第二层级**：使用带括号的阿拉伯数字，如 `(1) `、`(2) `、`(3) `
* **第三层级**：使用带填充效果的大方形项目符号 `■ `
* **第四层级**：使用箭头项目符号 `➔ `
* **第五层级**：使用带填充效果的大圆形项目符号 `● `

## 4. 内容细化程度 (Content Detail Level)
为了使其他人员能直观快速地读懂，PRD 的内容必须足够丰富和细致。请参考以下标准进行细化：

* **需求背景与场景**：
  * **背景**：详细描述当前业务现状、存在的问题（痛点）以及解决这些问题的必要性。
  * **场景**：列举具体的业务场景（如：场景一：XXX），描述在什么情况下，谁需要做什么，以达到什么目的。
  * **对象**：明确定义业务中涉及的核心对象及其包含的关键信息。
* **功能说明 (核心)**：必须按照界面的逻辑区域（如：全局布局与导航、查询筛选区域、数据列表区域、底部操作区域）进行结构化拆解。
  * **全局布局与导航**：必须明确描述侧边栏（左侧菜单树）、顶部导航/页签栏等全局元素的控件类型、包含的菜单项及交互功能（如：展开/折叠、多页签切换）。
  * **操作逻辑详述**：页面中所有的操作详情都需要详细描述，特别是操作的逻辑（如点击动作、校验规则、状态流转、数据带入规则、接口调用时机等）。
  * **弹窗内联描述**：所有操作的弹窗都需要详细描述，**绝对不能**使用“引用XXX页面的描述”或单独开辟一个“公共组件/弹窗”章节。弹窗的详细描述必须放置在触发该弹窗的对应操作部分，按照页面及页面中的操作顺序进行内联描述。
  * **功能完整性**：必须仔细核对代码或需求，确保不漏写任何功能（如导出、导入、批量操作、配置详情等）。
  * **界面元素描述**：对于每个区域内的每一个控件（输入框、下拉框、按钮等），必须详细说明：
    * **控件类型**（如：文本框、下拉选择、日期选择器、复选框等）
    * **标签文字/占位符**（如：标签文字："业务类型:"，占位符文本:"请输入..."）
    * **选项列表**（如下拉框的具体选项：全部、选项A、选项B）
    * **默认值**
    * **功能/点击动作**（如：支持多字段模糊搜索、执行筛选查询并重置页码为1）
    * **联动规则/禁用状态**（如：选择A时隐藏B，未选中行时禁用删除按钮）
  * **数据列表描述**：使用 **Markdown 表格**清晰定义列表的列信息，包括：序号、列标题、数据字段（英文驼峰）、说明（该列展示的内容或特殊标识，如颜色）。
  * **业务规则与异常处理**：明确列出该区域或功能的业务逻辑（如：各条件间是 AND 关系、默认排序规则）以及异常情况的 UI 表现（如：无数据时的提示语、必填项未填的错误提示）。
* **非功能性需求**：补充性能、安全、可用性、兼容性等方面的具体指标要求。

## 6. 禁止插入图片 (No Images Allowed)
* **禁止使用图片**：在生成的 PRD 文档中，**严禁**插入任何形式的图片（包括 Markdown 格式的图片 `![]()`、HTML `<img>` 标签或占位图链接）。
* **替代方案**：所有界面展示、流程图或逻辑关系必须通过**详细的文字描述**、**Markdown 表格**或 **Mermaid 流程图**（如果支持）来表达，确保文档纯文本化。

## 7. 内容真实性要求 (Content Authenticity Requirement)
* **严禁虚构内容**：生成的 PRD 文档内容必须与代码中的实际功能模块和逻辑严格一致。
* **基于事实描述**：对于代码中没有实现的功能、界面元素或业务逻辑，**绝对不能**进行虚拟、臆造或“补充”。如果在分析过程中发现信息不足，应以事实为准描述现有逻辑，或在文档中如实记录。
